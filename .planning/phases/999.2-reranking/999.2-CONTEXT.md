# Phase 999.2: Reranking - Context

**Gathered:** 2026-06-18
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase delivers a secondary reranking step in the RAG retrieval pipeline. After hybrid retrieval merges dense vector and BM25 results, a reranker re-scores and filters candidates using semantic relevance before they are passed to the LLM. In v1, we leave ports by defining the Reranker trait and implementing a pass-through NoOp.

</domain>

<decisions>
## Implementation Decisions

### Model Selection
- **D-01:** Implement a **Dual-Implementation** approach. Define a clean abstraction trait in Rust that supports both an external API (e.g., Cohere) and a local model (ONNX Runtime / Candle), defaulting to a Mock/NoOp in v1.

### Integration Architecture & Traits
- **D-02:** Define the Rust Reranker trait as an **async trait** (using `#[async_trait]` or native async fn) returning `Result<Vec<Chunk>, RerankError>`. This allows non-blocking remote API calls and offloads local CPU-intensive inference to `tokio::task::spawn_blocking`.

### Scoring & Candidate Selection (Budgets)
- **D-03:** Use **configuration-driven parameters**. The retriever fetches a large candidate pool (e.g., top-50) and passes them to the Reranker, which returns a smaller subset (e.g., top-5) to the LLM. In v1, the retrieval parameters will support separate fetch-counts and final-context-counts.

### Fallback & Degraded Modes
- **D-04:** Use **Soft Fallback**. If the reranker fails (timeouts, API rate limits, or model errors), log a warning and return the original hybrid-ranked candidates sorted by combined vector/lexical scores, enabling degraded mode execution.

### Agent Discretion
- **Discretion-01:** The agent has discretion over the exact error types in `RerankError` and how the local ONNX runtime thread pool is configured.

</decisions>
