# Rust Markdown Chunker & Token Estimation (Plan 2)

This plan implements the custom chunking engine in Rust, supporting both fixed-size and structure-aware chunking strategies, handles plain text and JSON ingestion safely, and integrates token estimation using `tiktoken-rs`.

<threat_model>
### Threat Model Config
ASVS Level: 1. Block on: high severity threats.

- **Threat:** DoS via Unbounded Recursion or Pathological Markdown Input.
  - **ASVS Control:** 12.4.1 (Verify that the application handles memory safely).
  - **Mitigation:** Use a safe parsing engine (`pulldown-cmark` is a robust, safe rust markdown pull parser) and limit recursive split depth for oversized block chunks (max recursion limit of 10).
- **Threat:** Parse Errors or Panic on Malformed JSON Input.
  - **ASVS Control:** 12.4.1 (Verify memory safety).
  - **Mitigation:** Plain text and JSON inputs are treated as raw text and chunked using the robust `fixed-size` strategy, avoiding parsing syntax asts of non-markdown files.
</threat_model>

## Tasks

### Task 1: Implement Fixed-Size and Structure-Aware Chunking Strategies
<read_first>
- `engine/Cargo.toml`
- `engine/src/chunker/mod.rs`
</read_first>
<acceptance_criteria>
- `engine/Cargo.toml` contains `pulldown-cmark = "0.10"` or similar.
- `engine/src/chunker/mod.rs` defines a `Chunk` struct containing:
  - `content`: String
  - `char_start`: usize
  - `char_end`: usize
  - `section_path`: Option<String>
- `engine/src/chunker/mod.rs` implements:
  - `chunk_markdown(text: &str, target_size: usize, overlap: usize) -> Vec<Chunk>` (Structure-aware strategy): Parses markdown headings using `pulldown-cmark` to track the heading hierarchy path (e.g. `/Setup/Installation`). Splits at block boundaries (headers, paragraphs, double newlines). If any block exceeds `target_size`, recursively splits it into character-windows with the specified overlap.
  - `chunk_fixed_size(text: &str, target_size: usize, overlap: usize) -> Vec<Chunk>` (Fixed-size strategy): Splits input text directly into sliding character windows of size `target_size` with the specified `overlap`. Sets `section_path` to `None`.
- **Ingestion Fallbacks**:
  - **Plain Text**: Lacking markdown structural tags, plain text run through `chunk_markdown` defaults to treating the document as simple paragraphs, gracefully falling back to character-window splits.
  - **JSON**: Treated as raw text and chunked strictly using the `chunk_fixed_size` strategy to prevent parser errors or JSON syntax corruption.
- Verification command: `cargo test -p engine chunker::tests` (test cases verifying markdown structure parsing, fixed-size windowing, JSON/Plain-text handling, and overlap bounds).
</acceptance_criteria>
<action>
Add `pulldown-cmark` dependency to `engine/Cargo.toml`.
Create `engine/src/chunker/mod.rs` implementing Markdown parsing, header hierarchy tracking, paragraph boundary splitting, and the fixed-size sliding character window chunker.
Create `engine/src/chunker/tests.rs` with unit tests verifying both chunking strategies, nested headers, overlap handling, plain-text fallback, and JSON raw text windowing.
</action>

### Task 2: Implement Token Estimation and Integrate Chunker into Background Worker
<read_first>

- `engine/src/chunker/mod.rs`
- `engine/src/main.rs`
</read_first>
<acceptance_criteria>

- `engine/src/chunker/mod.rs` defines `estimate_tokens(chunk: &str) -> i32` using the `o200k_base` singleton.
- Update background worker in `engine/src/main.rs` to:
  1. Extract metadata parameters: `chunk_strategy`, `chunk_size`, and `chunk_overlap` from the ingestion job request.
  2. Select strategy: Calls `chunk_markdown` if strategy is `structure-aware` (or empty default), and `chunk_fixed_size` if strategy is `fixed-size`. If the ingested file type is JSON, force `fixed-size` strategy.
  3. Invoke `estimate_tokens` on each chunk.
- Verification command: `cargo test -p engine`
</acceptance_criteria>

Update the sequential tokio worker loop in `engine/src/main.rs` to extract `chunk_strategy`, `chunk_size`, and `chunk_overlap` from the job request, call the chosen chunker function, and populate the estimated token counts.
</action>
