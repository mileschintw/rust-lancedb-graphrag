# Async Background Indexing & Status Polling (Plan 4)

This plan integrates the Markdown chunker, token estimator, OpenRouter API client, and LanceDB table operations into the Tokio background indexing loop, and wires up the status polling and database status updates in the Go Gateway.

<threat_model>
### Threat Model Config
ASVS Level: 1. Block on: high severity threats.

- **Threat:** Race Conditions during Concurrent Database Writes.
  - **ASVS Control:** 12.4.1 (Verify memory safety and thread safety).
  - **Mitigation:** Enforce a single-threaded sequential indexing consumer using a single worker tokio loop, protecting the write targets from race conditions and data corruption.
- **Threat:** File Path Traversal / Injection via Document IDs.
  - **ASVS Control:** 12.4.1 (Verify input validation on paths).
  - **Mitigation:** Ensure document IDs are strictly validated as UUIDv4 strings before passing them to LanceDB local storage queries, preventing file-system path manipulation.
</threat_model>

## Tasks

### Task 1: Complete Tokio background worker pipeline in Rust engine
<read_first>
- `engine/src/main.rs`
- `engine/src/db/mod.rs`
- `engine/src/client/mod.rs`
</read_first>
<acceptance_criteria>
- Rust background worker receives document ingestion jobs and performs:
  1. Sets in-memory registry status to `processing`.
  2. Invokes structure-aware chunker to obtain document chunks.
  3. Batch requests chunk embeddings from OpenRouter embeddings client (concurrency limit 5).
  4. Resolves entities using `ExactMatchResolver`.
  5. Performs database writes to LanceDB `nodes` and `edges` tables (saving metadata, chunk information, vectors, and placeholder summaries).
  6. Atomic upsert: if the `document_id` already exists, clears previous raw contents from the `documents` table and associated chunk nodes from the `nodes` and `edges` tables before writing new ones.
  7. Updates status to `completed` and records total chunk count (or set status to `failed` and records error message).
- Bounded channel worker implements a graceful shutdown receiver: on shutdown, allows the active processing document to finish indexing and write to database before exiting.
- Worker uses context-rich tracing spans with document ID.
- Verification command: `cargo test -p engine`
</acceptance_criteria>
<action>
Integrate the chunker module, embeddings client, and database manager inside the background worker task loop.
Hook a shutdown signal receiver (tokio select) into the main background worker queue pull loop, ensuring that if a shutdown event occurs, it completes any active index task currently running before returning.
Add `tracing::info_span!` scopes wrapping the chunking, embedding, and database write stages.
</action>

### Task 2: Implement polling endpoints and database updates in Go Gateway
<read_first>
- `gateway/main.go`
- `gateway/db/query.sql`
- `gateway/db/query.sql.go`
</read_first>
<acceptance_criteria>
- Go Gateway `GET /documents/{id}` checks the local PostgreSQL table for status.
- If status is `queued` or `processing`, calls `GetIngestionStatus` gRPC endpoint in Rust engine.
- If status returned is `completed` or `failed`, gateway executes a database transaction to update Postgres table with status, chunk count, and optional error message.
- End-to-end validation test script `verify-ingestion.sh` is provided in root directory to upload a sample file, poll status until completed, and verify database state.
- Verification command: `go test -v ./gateway/...`
</acceptance_criteria>
<action>
Update the HTTP router path `GET /documents/{id}` in `gateway/main.go` to support polling gRPC status and updating local metadata.
Create `verify-ingestion.sh` shell script executing a multipart curl file upload, polling the returned URL, and asserting completion status.
</action>
