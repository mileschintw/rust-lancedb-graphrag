# Ingestion Database & gRPC Scaffolding (Plan 1)

This plan implements the database metadata tables, Go Gateway multipart API routes, and Rust gRPC/Tokio background task queue scaffolding for the thinnest viable end-to-end ingestion path.

<threat_model>
### Threat Model Config
ASVS Level: 1. Block on: high severity threats.

- **Threat:** SQL Injection / Data Tampering in Go Gateway.
  - **ASVS Control:** 5.3.1 (Verify that the application uses parameterized SQL queries).
  - **Mitigation:** Use sqlc-generated pgx queries where all variables are parameterized.
- **Threat:** Resource Exhaustion via oversized file uploads.
  - **ASVS Control:** 11.1.2 (Verify that the application has a maximum limit for upload file sizes).
  - **Mitigation:** Enforce a strict 10MB limit in Go gateway HTTP upload handler.
- **Threat:** Ingestion Queue Exhaustion.
  - **ASVS Control:** 11.1.8 (Verify that application rate limiting is implemented to protect API endpoints).
  - **Mitigation:** Set Rust Tokio background queue capacity to 100 and reject additional inputs with `RESOURCE_EXHAUSTED` (HTTP 429).
</threat_model>

### Task 2: Implement End-to-End Ingestion Flow Scaffolding (Go & Rust)
<read_first>
- `proto/lancet/v1/lancet.proto`
- `gateway/main.go`
- `engine/src/main.rs`
- `engine/Cargo.toml`
</read_first>
<acceptance_criteria>
- `proto/lancet/v1/lancet.proto` includes `GetIngestionStatus` RPC and `GetIngestionStatusRequest` and `GetIngestionStatusResponse` messages.
- `buf generate` successfully updates Go and Rust proto bindings.
- Go gateway `POST /documents` endpoint validates file presence, inserts metadata with status `queued` in transaction, streams bytes in 64KB buffers to Rust `IngestDocument`, and returns HTTP 202.
- Rust engine `IngestDocument` stream handler stores the raw file in LanceDB `documents` table and enqueues to a bounded `tokio::sync::mpsc::channel` (capacity 100).
- If the bounded queue is full, Rust returns `Status::resource_exhausted`, mapped by Go gateway to HTTP 429.
- Rust background worker sequentially processes jobs: updates state registry to `processing`, logs indexing spans, mocks completions by changing state to `completed`, and records mock chunk counts.
- Go gateway `GET /documents/{id}` endpoint polls status from Rust engine if status in Postgres is not terminal (`completed` or `failed`), and updates Postgres status inside a database transaction.
- Verification command: `go test -v ./gateway/... && cargo test -p engine`
</acceptance_criteria>
<action>
Add the `GetIngestionStatus` endpoint and messages to `proto/lancet/v1/lancet.proto` and run `buf generate`.
Add `/config/config.toml` and `/config/config.dev.toml` defining dev environment configurations (db connection, port, openrouter api key, lancedb path).
Implement configuration loading in Go gateway using `viper` and in Rust engine using `config` crate.
Add `POST /documents` and `GET /documents/{id}` routes in `gateway/main.go`.
Add the gRPC client calls, status polling, and database status updates in `gateway/main.go`.
Initialize LanceDB table connection, status tracking registry (e.g. `Arc<DashMap<String, IngestionStatus>>`), Tokio channel, and background worker loop in `engine/src/main.rs`.
Implement graceful shutdown in the Rust worker (ensuring any actively processing task is finished before exit).
</action>
