# Phase 999.3: Query Reformulation Strategies - Context

**Gathered:** 2026-06-18
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase delivers advanced pre-retrieval query processing in the RAG pipeline. Before executing retrieval, the system expands the original query into multiple variants or generates hypothetical answers to maximize recall. In v1, we leave ports by defining the QueryReformulator trait and executing a pass-through NoOp.

</domain>

<decisions>
## Implementation Decisions

### Strategy Selection
- **D-01:** Implement a **Dual-Strategy** that supports both HyDE (for conceptual questions) and Multi-Query Expansion (for search/keyword queries), selected by configuration or runtime query analysis.

### Integration Architecture & State Machine Ports
- **D-02:** Define a dedicated **`QueryReformulator` trait** in Rust with an async `reformulate` method returning `Vec<String>`. In v1, the NoOp implementation will return a single element containing the original query unchanged.

### Latency & LLM Call Optimization
- **D-03:** Run LLM expansion calls concurrently using **`tokio::spawn`** (or `futures::future::join_all`). Support routing expansion tasks to a faster, lightweight local LLM (e.g. Llama-3-8B) or using structured JSON mode to minimize latency.

### Merging & Deduplication of Results
- **D-04:** Use **Reciprocal Rank Fusion (RRF)** to score, merge, and rank the candidate document chunks returned from parallel retrievals, followed by chunk-level deduplication.

### Agent Discretion
- **Discretion-01:** The agent has discretion over the default RRF constant parameter ($k \approx 60$) and how the deduplication merges chunk source citations.

</decisions>
