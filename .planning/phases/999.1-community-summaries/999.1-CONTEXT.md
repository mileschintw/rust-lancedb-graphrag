# Phase 999.1: Community Summaries (Global Graph Summarization) - Context

**Gathered:** 2026-06-18
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase delivers a pre-computed, hierarchical community summary layer over the GraphRAG knowledge graph, enabling global queries that summarize high-level themes across the entire document corpus. It is designed as a deferred backlog item, but leaves specific "extension ports" in the v1 implementation to make future integration seamless.

</domain>

<decisions>
## Implementation Decisions

### Community Detection Algorithm
- **D-01:** Partition the graph using the **Leiden Algorithm** to handle hierarchical communities.
- **D-02:** Support a multi-level hierarchy (leaf communities up to root). To support this in future versions, leave a `community_ids` integer array field in the v1 LanceDB node table.
- **D-03:** Run community detection natively in-engine as a **Rust service** (rather than offline Python scripts) to keep the data-plane execution unified.
- **D-04:** Trigger re-clustering asynchronously in the background using a **delta-threshold approach** in the Rust engine, tracking dirty nodes added/removed since the last partition run.

### Community Summary Prompting
- **D-05:** Feed both **nodes (entities) and edges (relations)** belonging to a community into the LLM prompt to generate summaries with complete topological context.
- **D-06:** Store community summaries in **structured JSON** containing a title, summary text, list of key entities, and findings/evidence.
- **D-07:** Generate **vector embeddings** for the JSON summaries to support semantic vector queries over communities.
- **D-08:** Handle context limit overflow in large communities via **topological sub-chunking** (splitting into sub-graphs, summarizing each, and merging).

### Global Query Execution
- **D-09:** Retrieve relevant communities using **hierarchical selection** (start at root level, identify relevant branches, and traverse down to leaf communities if detailed context is needed).
- **D-10:** Execute global query traversal/map-reduce orchestration logic in the **Rust RAG engine (data plane)** to minimize Go-to-Rust gRPC latency.
- **D-11:** Stream intermediate progress events (e.g. "Analyzing community A...") via a **gRPC server stream** to the Go gateway, followed by the synthesized answer tokens.
- **D-12:** Run in **Degraded Mode** if a sub-community synthesis fails (timeouts, LLM rate limits), continuing the query with successful nodes and flagging the missing context in citations.

### Ingestion & Update Strategy
- **D-13:** Update community summaries **incrementally** by comparing the Jaccard similarity of new vs. old memberships, only re-summarizing if the membership drift crosses a threshold.
- **D-14:** Store community summaries and hierarchical metadata in a separate dedicated **`communities` table** in LanceDB. In v1, we will define this schema (columns: `id`, `name`, `level`, `parent_id`, `summary_json`, `summary_vector`) as an empty placeholder schema in the database initialization code.
- **D-15:** Add a **placeholder metric** (e.g., `global_faithfulness` or `global_coverage`) in the v1 offline evaluation script returning a simulated or skipped score to make room for full GraphRAG evaluation later.
- **D-16:** Re-run partitioning and summarization (**invalidate-and-rebuild**) automatically if community detection parameters (e.g. Leiden resolution) are modified in configuration.

### Agent Discretion
- **Discretion-01:** The agent has discretion over the choice of Leiden algorithm crate/library and native bindings in Rust.
- **Discretion-02:** The agent has discretion over the exact JSON schema fields for findings/evidence in the community summary prompt.

</decisions>
